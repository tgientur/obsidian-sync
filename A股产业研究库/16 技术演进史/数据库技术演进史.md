# 数据库技术演进史

## 技术起源

数据库管理系统的萌芽可以追溯到1960年代——计算机从科学计算走向商业数据处理，需要持久化、结构化地存储和检索大量数据。在此之前，数据存储在磁带、打孔卡片或文件中，程序直接管理存储格式，每次存储格式变化都需重写代码。

1968年IBM发布IMS(Information Management System)，成为第一个主流商用数据库系统。IMS采用层次模型——数据组织为树形结构(父-子关系)，查询通过遍历树节点进行。这种模型对"一对多"关系(如一个订单对应多个条目)自然高效，但对"多对多"关系(如学生和课程)处理困难。

几乎同时，1969年Charles Bachman领导开发的IDS(Integrated Data Store, GE)推广了网状模型——记录之间可以通过SET关系形成有向图，允许更灵活的关系建模。CODASYL(数据系统语言会议)在1971年发布了网状数据库的标准规范。

层次和网状数据库的共同问题：数据访问路径是预定义的——查询必须从根遍历，变更数据结构需要同时修改全部应用程序。这催生了关系模型的诞生。

## 第一代方案：层次/网状数据库 (1960s-1970s)

### 层次数据库：IBM IMS

IMS基于DL/I(Data Language/I)接口，数据以Segment(段)组织成多层树结构。通过PSB(Program Specification Block)定义程序的数据视图。IMS是当时最大银行和航空系统的核心——至今仍在全球许多大型银行的CICS/IMS核心系统中运行。IMS Transaction Manager每天处理数十亿笔交易。

### 网状数据库：IDS/CULLINET

Cullinane公司的IDMS(Integrated Database Management System)是网状数据库最成功的商业实现，支持COBOL调用和IDD(集成数据字典)。在1970年代末至1980年代初，IDMS是IBM主机上的主流数据库之一。网状模型的优势是可以直接表达复杂关系，但查询(导航式)需要程序员手动跟踪set指针，开发效率极低。

### 层次与网状数据库的核心缺陷

二者都属于"导航式数据库"(Navigational Database)——用户必须知道数据如何组织和连接，需要手动遍历结构找到所需记录。这个缺陷在数据量增长和查询需求复杂化后变得不可接受。此外，数据与结构紧耦合——增加一个字段或建立新关系，应用程序代码几乎需要全部重写。

## 第二代方案：关系型数据库 (1970s-2000s)

### 关系模型的理论奠基

1970年，IBM的Ted Codd发表论文"A Relational Model of Data for Large Shared Data Banks"，正式提出关系模型：数据以关系(表)表示，用元组(行)和属性(列)组织，通过关系代数(选择、投影、连接等)操作数据。关系模型的核心贡献是将数据访问与物理存储分离——用户只需要声明"想要什么"（SQL查询），而不需要知道"怎么找"（物理路径）。

1974年IBM的System R项目验证了关系模型的可行性并发明了SQL(结构化查询语言，最初称为SEQUEL)。SQL借鉴了关系代数和关系演算，成为数据库交互的语言标准。

### Oracle与SQL时代的开启

1977年Larry Ellison、Bob Miner和Ed Oates创建SDL(后更名为Oracle)。Oracle 2(1979)是第一个商用SQL关系型数据库，运行在VAX/VMS和Digital UNIX上。1983年Oracle 3用C语言重写，可移植到不同操作系统。1986年Oracle 5引入C/S架构和分布式查询。

1980年代SQL成为国际标准(SQL-86, SQL-89, SQL-92, 1999年SQL:1999支持面向对象扩展)。关系数据库成为企业IT基础设施的标配。

### 主要代表产品

**Oracle**(1979起)：企业级数据库的标杆。支持RAC(Real Application Clusters, 2001)—多个服务器共享访问同一数据库。Oracle 10g/11g引入自动存储管理(ASM)、Data Guard灾备、自动工作负载仓库(AWR)。2020年代Oracle 23c支持JSON、图、向量搜索的融合数据库(Converged Database)概念。

**IBM DB2**(1983起)：IBM大型机(System/390)上的DB2是第一个在主机上实现关系模型的商业产品。1989年DB2扩展到LUW(Linux/Unix/Windows)。DB2 for i5/OS(AS/400)以其高稳定性和与IBM硬件的高度集成在银行核心系统上长期使用。

**Microsoft SQL Server**(1989起)：与Sybase合作开发，后在Windows NT平台独立发展。SQL Server 6.5(1996)首次支持企业级功能，7.0(1998)完全重写引擎。2005年SQL Server 2005引入CLR集成、DMV动态管理视图、Database Mirroring。2011年Denali引入AlwaysOn可用性组。SQL Server在Windows生态的中端市场占据绝对优势。

**MySQL**(1995起，Monty Widenius)：开源数据库的标杆。InnoDB存储引擎(由Innobase Oy开发，2005年被Oracle收购)提供事务支持(ACID)和外键约束。在Web应用(LAMP架构)中成为标准。2008年被Sun收购，2010年随Sun被Oracle收购。社区版与Oracle的对比中保持GPL协议分支(如MariaDB)。

**PostgreSQL**(1996起，源自加州伯克利Postgres项目)：最先进的开源关系数据库。支持MVCC(多版本并发控制，1996)、窗口函数、CTE(Common Table Expressions)、表继承、自定义类型/函数、PostGIS地理空间扩展。在OLAP、GIS、时序等领域都有广泛应用。2020年代凭借出色的扩展性和新功能(逻辑复制、并行查询、原生分区)获得快速增长。

### 关系数据库的核心技术

- **ACID**：Atomicity(原子性)、Consistency(一致性)、Isolation(隔离性)、Durability(持久性)。Jim Gray等人在1970-80年代定义了事务的理论基础。Oracle/MVCC(Snapshot Isolation)和SQL Server(锁+乐观隔离)的实现方式不同。
- **B+树索引**：Rudolf Bayer和Ed McCreight在1972年发明，所有关系数据库的核心索引结构。B+树的扇出因子(每个节点存储的记录数)高，深度通常在2-4层，即使是数十亿行的表也只需要3-4次I/O读。
- **查询优化器**：基于代价的优化器(CBO, Cost-Based Optimizer)在System R中首次实现。优化器将SQL转为关系代数树，考虑多种连接顺序、索引选择、连接算法(Nested Loop/Hash Join/Sort-Merge Join)，选择估算成本最低的执行计划。

## 第三代方案：NoSQL与新SQL (2000s-2010s)

### 互联网爆发与关系数据库的局限

2000年代互联网公司的数据量(Web索引、用户行为、社交媒体)达到PB级别，关系数据库的以下问题暴露出来：
- **水平扩展困难**：关系数据库的ACID事务和JOIN操作天然适合垂直扩展(加硬件)，但跨节点水平分片(Shared Nothing)的工程实现极其复杂。
- **灵活Schema限制**：预定义Schema在快速迭代的互联网产品中成为瓶颈。
- **写入吞吐量瓶颈**：B+树需要顺序写入，随机写入性能差。对于每秒数万条写入的日志、点击流场景，关系数据库难以承受。

### NoSQL运动

2009年，NoSQL(Not Only SQL)的概念由Johan Oskarsson在旧金山的聚会上正式提出。NoSQL数据库放弃了关系模型和ACID事务，换取水平扩展能力和灵活的数据模型。分类：

**键值存储(Key-Value Store)**
- **Redis**(2009, Salvatore Sanfilippo)：纯内存数据库，支持字符串、哈希、列表、集合、有序集合、Bitmaps、HyperLogLog、流等多种数据结构。单机延迟微秒级，支持主从复制和Sentinel/Cluster高可用。典型场景：缓存、会话管理、实时排行榜(如微博热搜)。在2020年代Redis Stack扩展了JSON、搜索和图模块。
- **Amazon DynamoDB**(2012)：基于Dynamo论文的完全托管NoSQL服务。SSD存储，微秒级延迟。关键特性：自动分片(Partitioning)、最终一致/强一致可选、DynamoDB Streams用于流处理、DAX加速层。
- **RocksDB**(2013, Facebook)：基于LSM-Tree的嵌入式键值存储引擎，底层采用LevelDB优化写放大。广泛应用于存储引擎层(MyRocks、TiKV、RMDB)。

**文档数据库(Document Store)**
- **MongoDB**(2009, 10gen公司)：面向文档的NoSQL数据库，数据以BSON格式存储。文档是Schema-less的JSON-like结构。支持复制集(Replica Set)和自动分片(Sharding)。聚合框架(Aggregation Pipeline)提供了基于文档的数据分析能力。MongoDB 4.0(2018)引入多文档事务(ACID)。2023年MongoDB Atlas(云托管服务)占营收的70%+。
- **Couchbase**(2011, CouchDB和Membase合并)：结合文档存储和键值缓存。支持N1QL(SQL-like查询)。Couchbase Server 7.0(2021)引入基于范围的索引扫描、列存储(Index Service优化分析查询)。

**列式存储(Wide-Column Store)**
- **Apache HBase**(2008, BigTable的开源实现)：运行在HDFS之上，支持稀疏、链表格式的数据存储。Row Key排序存储，Region自动拆分。用于实时随机读写超大规模数据集。2010年代被Facebook用于消息系统(每天千亿条消息)。2020年代被TikTok/字节跳动的Abase等逐步替换。
- **Apache Cassandra**(2008, Facebook开发, 2009开源)：基于Amazon Dynamo和Google BigTable的混合设计。无主(Leaderless)架构——每个节点平等，任何一个节点可以写入。数据分布通过一致性哈希。Cassandra在Netflix、Apple(iCloud)等大规模场景下验证了超大规模写入的能力。Cassandra 4.0(2021)改进虚拟机(Java)优化了垃圾回收。

**图数据库(Graph Database)**
- **Neo4j**(2007)：原生图存储——节点和关系在磁盘上直接存储为图结构，关系遍历操作O(1)常数时间(而非JOIN的O(n))。使用Cypher声明式查询语言。在社交网络、推荐引擎、欺诈检测场景下优势明显。Neo4j 5.x(2022)支持分片和基于属性的访问控制。
- **Amazon Neptune**(2017)：托管图数据库服务，支持Property Graph和RDF两种模型，兼容Gremlin和SPARQL查询。

### NewSQL与分布式数据库

NoSQL解决了水平扩展但牺牲了事务和关系查询。NewSQL的追求：保持NoSQL的扩展性，同时提供关系数据库的ACID事务和SQL接口。

**Google Spanner**(2012发表论文, 2017公开)：第一个全球分布式关系数据库，使用TrueTime API(TFTP服务器+PTP硬件时钟)同步时钟，在所有节点上实现外部一致性(external consistency)。Spanner是NewSQL的标杆——证明全球一致性的关系数据库在PB级别同时支持强一致事务和高可用是可行的。Spanner的架构影响了一大批后续分布式数据库。

**TiDB**(2015起, PingCAP)：开源NewSQL分布式数据库。架构设计借鉴Spanner和Percolator：
- **TiKV**：底层分布式KV存储，基于Raft共识协议(每个Region有3个Replica)，RocksDB引擎。水平弹性扩展(通过PD调度Region Split/Merge)。
- **TiDB Server**：SQL层，解析MySQL协议，处理查询优化和分布式执行。TiFlash列存引擎提供HTAP能力。
- TiDB 6.x(2022)引入全局内存表(Memory Table)和小表缓存；TiDB 7.x(2023)提升弹性扩展和资源隔离。

**CockroachDB**(2015起)：受Spanner启发，也使用Raft共识协议，但时钟同步基于HLC(混合逻辑时钟)，不需要TrueTime的硬件支持。CockroachDB的架构核心是：所有数据自动分片(64MB Range)，每个Range在至少3个节点间复制。

**OceanBase**(2010起, 蚂蚁集团/阿里)：自研分布式数据库。核心创新：原生的LSM-Tree引擎+多副本Paxos一致性+SQL层透明分库分表。
- OceanBase 1.0(2014): 基础分布式能力，支持SQL+事务，3500行以内的SQL兼容MySQL协议。
- OceanBase 2.0(2019): 添加Oracle兼容模式，存储过程、序列、触发器，为金融核心迁移做准备。在TPC-C基准测试中以6088万tpmC击败Oracle，成为全球第一(因涉嫌参数争议也有讨论，但性能指标客观)。
- OceanBase 3.0(2021): 支持多租户，提升AP性能，引入向量化执行引擎。
- OceanBase 4.0(2022): 700行核心代码+4.0版本引入"单机分布式一体化"——一个数据库既能在单台128GB服务器运行，也能扩展到数百节点。核心改进：自适应日志流、内存池化、多模引擎(混合行存/列存)。
- 2023年OceanBase 4.2引入向量数据库能力。

**国产数据库：达梦(DM Database)**
达梦数据库(武汉达梦，前身华中科大的数据库研究室，1980年代开始研制)是中国关系数据库自主化时间最长的产品。达梦DM8(2019)兼容Oracle语法，支持MPP并行查询和读写分离集群(DSC+DW)。DM9(2023)增强分布式能力、共享存储集群功能。达梦在党政信创的数据库采购中占据第一位，2023年在中国关系型数据库市场中市占率约5%。

### 内存数据库

除Redis外，SAP HANA(2010)是一种列式内存关系数据库——所有数据在主存中，磁只用于持久化。HANA将数据压缩后加载到内存，缓存友好的列式存储实现分析查询1000倍加速(In-memory computing)。HANA是SAP核心产品S/4HANA的唯一数据库支撑。SAP HANA云版(2021+)支持多租户和Kubernetes部署。

## 当前主流方案：云原生数据库与AI融合 (2020s)

### 云原生数据库

**Amazon Aurora**(2014, 2016 GA)：MySQL/PostgreSQL兼容，存储与计算分离。Aurora的存储层是一个具备跨3个AZ的6副本分布式存储——写入数据先写入存储层(Quorum，6副本中4个确认)，然后返回成功。Aurora将Redo Log下沉到存储层，只有redo log在网络上传输，大幅减少了网络I/O量。Aurora支持50Gbps带宽、128TB容量上限、灾备到多个Region。

**Google AlloyDB for PostgreSQL**(2021)：运行在Google Cloud上，存储层基于Google Colossus分布式文件系统。AlloyDB自适应引擎将数据在内存中缓存为列式格式，分析查询比标准PostgreSQL快4倍。关键特性：Machine Learning辅助的自动索引、Index Advisor。

**AWS Redshift Spectrum**(2017)：查询存储在S3上的数据，不加载到本地。Redshift Spectrum是数据湖(Data Lake)查询的代表——存储(S3)+计算(Redshift集群)分离，按查询的数据量计费。

### Serverless与存算分离

Serverless数据库的终极形态：数据库完全托管，按计算和存储分别计费，支持从0到峰值自动伸缩(Autoscaling Scale-to-Zero)。代表：
- **AWS Aurora Serverless v2**(2022)：0.5 ACU起步，最多256 ACU。写入秒级扩展，读取亚秒级水平扩展。
- **Azure Serverless SQL DB**(2017)：按秒计费，自动暂停/恢复(1分钟不活动挂起)。
- **PlanetScale**(2018, 基于Vitess)：提供MySQL兼容的无服务器数据库。

### AI与数据库的融合

2020年代数据库进入了AI原生阶段：
- **向量数据库(Embedding Vector Database)**：专为AI Embedding(嵌入向量)设计的存储引擎。Pinecone(2019)、Weaviate(2016)、Qdrant(2020)、Milvus(2019, Zilliz)是主要玩家。通过近似最近邻(ANN, Approximate Nearest Neighbor)搜索算法(HNSW、IVF_PQ)实现毫秒级的语义向量检索——这是RAG(检索增强生成)应用的核心基础设施。
- **AI驱动的数据库优化**：Oracle 23c利用ML为SQL作业自动推荐索引、压缩策略。SQL Server 2017+的Query Store自动识别和回滚回归查询。腾讯TBase的智能参数调优利用强化学习自动调整数据库参数。
- **NL2SQL**：自然语言转SQL。LLM(如GPT-4、Llama 3)将用户问题自动生成SQL查询，使非技术人员也能与数据库交互。代表产品：Weaviate的LLM集成、DB-GPT(开源框架)。
- **AI for Database内部运算**：基于学习的查询优化器(如PostgreSQL的Learned Cardinality Estimation)、自动索引选择(AI自动推荐最佳索引组合)、自适应锁管理器(预测并发冲突提前升级锁)。

## 未来技术路线

### 融合数据库(Converged / Multimodal Database)

Oracle 23c提出单一数据库支持关系型、文档(JSON)、图、空间、向量、文本搜索(Hybrid Search)。这种"一站式"方案避免了为不同数据模型部署多个数据库的运维开销和数据同步问题。PostgreSQL通过扩展(PostGIS、age、pgvector)已经成为事实上的融合数据库基础。

### Database Reliability Engineering (DBRE)

SRE(站点可靠性工程)方法论在数据库运维中的深化：混沌工程(随机注入数据库节点故障，验证系统容错如JanusGraph)、自适应调优(自动检测性能退化并回滚变更)、数据验证引擎(持续验证主从数据一致性)。

### 存内数据库(Disaggregated Memory)

CXL(Compute Express Link)协议的普及将使内存从计算节点分离，通过CXL.mem和CXL.io协议共享。数据库可以在N个节点间共享一个庞大内存池，彻底消除传统分布式数据库中的缓存一致性开销。SAP HANA的CXL变体正在研发中。

### 数据湖仓一体(Lakehouse Architecture)

Databricks(2013, Apache Spark)商业化的Lakehouse概念——数据仓库的ACID事务和Schema管理直接构建在对象存储(如S3/ADLS)的开放格式(Delta Lake/Iceberg/Hudi)之上。不再需要"ETL到数据仓库"——直接在数据湖上跑分析查询。Delta Sharing(2022)统一了跨平台的数据共享协议。

## 核心瓶颈

### 分布式事务的几何级复杂度

Paxos/Raft一致性协议每秒只能容忍有限数量的领袖变更(Leader Election)。2PC(两阶段提交)在跨节点事务中性能下降显著——协调者单点瓶颈、阻塞协议在故障时锁持有资源。写放大：Raft在3副本时每笔写入至少复制3次，对于TB级数据库，硬件的网络I/O是主要成本。跨AZ延迟(典型2-5ms)是地理分布式数据库必须面对的基础物理瓶颈。

### 写入与查询的根本矛盾

行存储(OLTP)优化点查和随机写入，列存储(OLAP)优化大扫描和聚合。单一架构同时优化两种负载的HTAP(Hybrid Transactional/Analytical Processing)在工程上极其困难——写放大和读冷却需要存储引擎在两种格式间转换。TiDB的TiFlash(列存副本)、Oracle In-Memory Column Store、SQL Server Columnstore Index都是妥协方案。

### AI对数据治理的挑战

LLM生成的数据质量参差不齐，将LLM输出持久化到数据库中带来了数据完整性、合规性(如GDPR中"解释权")、偏见放大等问题。向量数据库在10亿级向量上ANN搜索的召回率在99%以上，但一致性模型弱(最终一致性)——无法保证刚写入的向量立刻可检索。

## 产业影响

### 全球数据库市场格局

2023年全球数据库市场规模约900亿美元(Gartner)。Oracle占市场领导者地位(约25%)，Microsoft SQL Server(约20%)，IBM Db2(约5%)。开源数据库(MySQL、PostgreSQL、MongoDB)的份额逐渐扩大。云数据库(AWS RDS/Aurora、Azure SQL DB、GCP Cloud SQL)成为增长最快的细分市场，云数据库的TCO在同等规模下通常低于自建数据库的50%。

### 中国数据库市场与信创

中国数据库市场规模约500亿元人民币(2023，IDC)。政策驱动"信创"替换——要求政企在2027年前逐步替换Oracle/DB2等国外数据库。竞争格局分为三条赛道：
- **公有云数据库**：阿里云(阿里RDS+ PolarDB)、华为云(GaussDB)、腾讯云(TDSQL)。PolarDB(2020)的存储计算分离架构支持100T数据量和100万QPS。
- **闭源商业数据库**：OceanBase(蚂蚁集团，金融行业市占率很高，2023年营收约5亿元)、达梦(DM8，党政信创最大份额)、人大金仓(KingbaseES，基于PostgreSQL)、南大通用(GBase)。
- **开源分布式**：TiDB(PingCAP，2022年营收约3亿元，海外占比超30%)。2024年TiDB推出Serverless Tier支持自动暂停恢复。
- **华为GaussDB**(2021起)：华为全栈自研分布式数据库(CloudNative+Share-Nothing架构)，底层存储基于华为自研分布式存储(DAAS)。GaussDB在2023年替换了部分中国工商银行的Oracle核心系统。

### 关键技术参数对比

| 产品 | OLTP TPS(Oracle等效) | 节点扩展 | 事务隔离 |
|------|------|------|------|
| Oracle 23c | 基准线 1x | 最大32节点(RAC) | SI+Serializable |
| OceanBase 4.x | 0.8-1.1x | 1000节点 | Read Committed+SI |
| TiDB 7.x | 0.6-0.8x | 100+节点 | SI+RC |
| PolarDB | 0.7-0.9x | 计算分离 | RC+SI |
| MySQL 8.0 | 0.3-0.5x(单机) | MGR/Group Replication | RC+SI |

### 替换迁移动力

金融行业是国产数据库替换的主战场：银行核心系统(存款、贷款、支付)对一致性和可用性要求最高(RPO=0, RTO<30s)，替换风险最大。通常在1-2年内分步替换：先替换外围(数据仓库、报表系统)，再替换核心。2021-2023年已有数家银行(如中国工商银行在GaussDB、网商银行在OceanBase)完成了核心系统替换。
